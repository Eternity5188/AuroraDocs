# AuroraDocs 系统架构

## 1. 目标架构

AuroraDocs 采用“后端远程GPU服务 + 前端轻量客户端”的现代架构：

- 后端部署在 AutoDL / 校园 GPU 服务器（如单卡 NVIDIA RTX 4090）
- 前端运行在 Windows 或 Web 浏览器，作为交互层和任务调度入口
- 客户端通过 REST API / WebSocket 与后端通信，后端负责所有训练和推理计算

## 2. 架构组件

### 2.1 后端服务

- **FastAPI**：提供样例管理、训练任务提交、模型管理、推理接口等 API
- **Celery + Redis**：作为异步任务队列，调度 LoRA/QLoRA 微调任务
- **vLLM + DeepSpeed**：在 GPU 上执行高性能推理和训练
- **PostgreSQL**：存储用户、任务、模型元数据
- **MinIO**：存储样例文档、训练数据和生成结果
- **Milvus**（可选）：向量检索与相似度检测

### 2.2 前端客户端

- **Vue3 / Electron**：可作为 Windows 桌面客户端或 Web 管理界面
- **作用**：文件上传、任务管理、进度展示、结果浏览，以及用户自定义 API 配置
- **用户配置**：在客户端保存用户自己的 API 选择、终端地址和密钥，避免后端直接保留个人私有信息，以及用户自定义 API 配置
- **用户配置**：在客户端保存用户自己的 API 选择、终端地址和密钥，避免后端直接保留个人私有信息
- **限制**：不直接执行模型训练，不承载大模型运行环境

### 2.3 通信与安全

- **通信协议**：HTTPS / WebSocket
- **认证方式**：JWT 或 Token
- **资源隔离**：训练任务通过后端统一调度，避免客户端暴露 GPU 细节
- **数据隐私**：样例和模型在后端集中管理，可选择私有部署

## 3. 现代化原则

- **职责分离**：训练与推理留给后端，客户端只负责交互
- **轻量接入**：客户端支持 Windows 与 Web，适合学生和企业用户
- **可扩展性**：后端可从单GPU扩展到多GPU或云端集群
- **高可用**：任务队列和缓存系统保证训练与推理稳定
- **可迁移性**：后端容器化部署，便于迁移到不同 GPU 平台

## 4. 推荐部署方式

1. 使用 AutoDL 或校内 GPU 服务器部署后端服务
2. 后端容器化：Docker Compose 或 Kubernetes（可选）
3. 前端作为独立客户端发布，连接后端 API
4. 训练任务采用 LoRA/QLoRA 微调，降低 GPU 资源消耗
5. 未来可扩展为“Web 入口 + 远程训练集群 + 多客户端”模式
